\title{Sparse Temporal Test-Time Training for V-JEPA 2.1}
\author{Burn JEPA Contributors}
\date{Preprint draft, May 2026}

\begin{document}
\maketitle

\begin{abstract}
V-JEPA 2.1 learns strong dense video features, but dense tubelet
patchification spends computation on every pixel and token in a clip.  We ask
whether a video stream that is already spatially sparse, for example from
AutoGaze, can enter V-JEPA 2.1 without dense pixel patchification while still
retaining useful temporal context.  We evaluate a Burn-native sparse V-JEPA 2.1
pipeline that loads the official ViT-B checkpoint, skips masked pixels during
GPU patchification, and trains small encoder-side test-time training (TTT)
adapters so a sparse per-frame student approximates a frozen 3D V-JEPA teacher.
The implementation applies all 312 mapped checkpoint tensors and matches the
upstream \texttt{torch.hub} model within $2.92\times10^{-5}$ max absolute error
on sparse encoder and predictor parity checks.  A first sparse TTT student
improves 16-frame held-out free-run loss from 0.4544 to 0.3021, but
long-rollout probes reveal fast-state decoherence at 64 frames.  Adding
TBPTT-style carried state, detach boundaries, state decay, state/update
regularization, scheduled resets, and a sequence-length curriculum stabilizes
32--64 frame rollouts: a 1024-step CUDA run carries state on 604 of 1024
windows and reduces 64-frame loss from the prior 1.70--1.97 range to
0.384--0.385.  The sparse inference path is already fast, reaching 510
rolling windows/s at 224px/1\% density and 90 windows/s at 720p/25\% density;
the main remaining bottleneck is TTT backward/optimizer cost and larger-scale
open-set validation.
\end{abstract}

\section{Introduction}

V-JEPA 2.1 is a joint-embedding predictive architecture for image and video
self-supervision \cite{mur2026vjepa21}.  Its video path is naturally dense:
for a 16-frame 224px clip with 16px patches and tubelet size 2, the encoder
sees 1568 video tokens before masking.  Streaming perception pipelines often
already know that only a subset of image regions matters, for example from
AutoGaze or frame-difference attention.  The question in this report is
whether V-JEPA 2.1 can use those sparse regions directly while retaining some
of the temporal information normally supplied by dense 3D convolutional
tubelets.

We study this as a systems and distillation problem rather than a claim of a
finished production model.  The contribution is a measured implementation:
Burn-native V-JEPA 2.1 parity, sparse positional encoding, sparse GPU
patchification, sparse TTT rollout, training/evaluation reports, and targeted
ablations over masks, encoder adapter placement, backend throughput, and
long-horizon fast-state health.

The experiments are organized around five questions.  First, does the Burn
model reproduce the official V-JEPA 2.1 sparse encoder and predictor?  Second,
does a sparse per-frame student learn useful teacher features at all?  Third,
does its recurrent fast state remain stable when windows are rolled forward in
time?  Fourth, do mask choices such as AutoGaze and patch difference change
quality or cost?  Fifth, does sparsity reduce runtime in the intended places,
and where does training still spend time?  This framing is important because a
single scalar loss cannot answer all five questions.

\section{Method}

\subsection{Sparse V-JEPA Forward}

Let $V\in\mathbb{R}^{B\times C\times T\times H\times W}$ be a video window and
let $G$ be the V-JEPA tubelet grid.  A context mask
$M_c\subseteq G$ selects visible tokens and a target mask $M_t\subseteq G$
selects held-out target tokens.  The sparse forward computes
\begin{equation}
  z_c = E_\theta(V, M_c), \qquad \hat{z}_t = P_\theta(z_c, M_c, M_t),
\end{equation}
where $E_\theta$ is the V-JEPA encoder and $P_\theta$ is the V-JEPA predictor.
Sparse plans cache token indices, sequence order, coordinates, and positional
encoding tensors.  The WGPU and CUDA sparse-patchify paths use
\texttt{burn\_flex\_gmm} so masked pixels are skipped before patch embedding
instead of gathered after dense patchification.

\subsection{Temporal TTT Student}

The student is a sparse single-frame rollout with recurrent TTT state.  Each
selected encoder block receives a zero-initialized residual adapter with a fast
state $F$.  For hidden tokens $X$, the adapter returns
\begin{equation}
  X + W_o(XF/\sqrt{d}),
\end{equation}
then updates $F$ from chunked hidden states.  With zero-initialized output
projection, an untrained adapter is initially function-preserving.  During
deployable evaluation the fast state is updated from detached student hidden
states, not from teacher tokens.  Teacher-forced evaluation remains available
only as a diagnostic.  This follows the broad TTT idea of adapting fast weights
from stream-local signals \cite{sun2019ttt,feng2026inplace}.

The primary objective is feature distillation from a frozen 3D teacher:
\begin{equation}
  \mathcal{L}_{feat}
  = \left\| \Pi_M S_\psi(V, M_c) - \Pi_M E_\theta(V) \right\|_2^2,
\end{equation}
where $\Pi_M$ selects the sparse or diagnostic token set.  An optional predictor
loss remains useful for standard JEPA pretraining, but it is not the TTT adapter
objective in this work.

\paragraph{Encoder-only adapter placement.}
We place TTT modules only in the encoder.  The production artifact is an encoder
that emits sparse, temporally adapted patch features for downstream clients.
Adding TTT to the predictor would adapt a training head rather than the shipped
representation, and it would require extra predictor-token updates after the
sparse encoder step.  That violates the intended efficiency boundary: masked
patches should be skipped before patchification and sparse token updates should
remain inside the encoder representation path.  Predictor-side TTT is therefore
out of scope for this paper.

\section{Experimental Protocol}

All V-JEPA 2.1 correctness claims use the official
\texttt{vjepa2\_1\_vit\_base\_384} checkpoint loaded from Meta's V-JEPA 2.1
\texttt{torch.hub} entry point.  Quality runs use local open-set video windows
at 224px with 16-frame student clips and a frozen 3D teacher.  Unless otherwise
stated, sparse masks keep 314 of 1568 context tokens (20.0\%) and 79 target
tokens (5.0\%).

Long-rollout diagnostics reuse the same trained adapter and evaluate 32--128
frame windows.  To separate temporal state drift from mask coverage artifacts,
we report both real precomputed AutoGaze masks and a deterministic
\emph{temporal-uniform} sparse mask that keeps the same 20\% context density
but spreads tokens evenly across tubelets.  These diagnostics are deliberately
not throughput benchmarks: they run extra reset, freeze, reverse, shuffle, and
segment probes and perform host-visible metric reads.

The stabilization run trains on ordered manifest streams rather than
independent windows.  TTT fast weights are carried across adjacent windows,
detached at each TBPTT boundary, decayed by 0.97, and regularized with
$10^{-6}$ state-norm and $10^{-5}$ update-norm penalties.  A scheduled reset
curriculum ramps from one 16-frame window to four windows, i.e. 64 frames, over
the first 512 optimization steps.

Backend labels are intentionally conservative.  Burn exposes both
\texttt{webgpu} and \texttt{wgpu} feature gates.  They are distinct Rust backend
types, but both sit in the WebGPU/wgpu backend family.  This paper therefore
groups them as the WebGPU-family lane except when a row specifically measures
the native WGPU \texttt{burn\_flex\_gmm} sparse-patchify extension.

\paragraph{How to read the experiments.}
The result tables are not interchangeable leaderboard rows.  They form a gate
sequence for a deployable sparse temporal encoder.  Numerical parity validates
that the Burn model is the same V-JEPA 2.1 computation before we interpret any
quality or speed result.  Sparse TTT quality tests whether the encoder-side
student works without teacher tokens at inference time.  Long-rollout probes
test whether the recurrent fast state remains bounded beyond the 16-frame
training window.  Mask-policy ablations ask whether a specific sparse driver,
such as AutoGaze, is better than generic sparse token selection at the same
density.  Throughput sweeps finally ask whether the system saves work in the
right place: pixel patchification and sparse token forward for inference, and
backward plus optimizer for training.

\subsection{Numerical Correctness}

The parity check is the entry criterion for the rest of the paper.  Sparse
patch selection changes token order, positional indexing, and predictor inputs;
a small training improvement would be meaningless if those details diverged
from the reference V-JEPA 2.1 module.

after training, can the sparse student run forward using only its own recurrent

Table~\ref{tab:main} answers the first modeling question: sparse encoder-side
TTT is useful under deployable free-run semantics.  The trained sparse student
improves sparse loss by 33.5\% relative to the zero-init adapter baseline.  The
full-grid row is included only to check that sparse training did not destroy
dense-token feature geometry; it is slower and should not be compared as a
deployment throughput number.  The streamed row answers a different question:
when adjacent 16-frame windows carry TTT state, quality remains close to the
independent-window result while actually exercising the stateful online path.

The stabilization run uses the same V-JEPA 2.1 checkpoint and real AutoGaze
masks, but trains with carried TTT state.  It runs 1024 CUDA steps at 0.528
samples/s, carries state on 604 windows, and reaches final/best training loss
0.2408/0.2348.  Backward and AdamW still dominate: 1675.9 of 1938.9 training
seconds are spent in backward plus optimizer.  This is why throughput
experiments later separate sparse forward inference from training-step cost.

metric rather than a secondary timing detail.

Table~\ref{tab:long} changes the production interpretation.  At 16 frames the
persistent state is useful: resetting or freezing the fast update is worse than
free-run persistence.  At 32 frames the state still helps overall, but the late
temporal third is already substantially worse than the early third.  By 64
frames, persistent state is worse than resetting each frame, both for
temporal-uniform masks and for real AutoGaze masks.  This is a direct
decoherence signal: the adapter has learned a useful short-horizon update, but
its fast state is not cleanly bounded over long streams.  The TBPTT rows show
that explicit long-horizon training largely fixes this failure mode at 32--64
frames: loss remains near 0.36--0.39 and late--early drift is small instead of
catastrophic.  This does not prove unbounded-stream stability, but it converts
the main negative result into a targeted training requirement.  The streamed
16-frame eval in Table~\ref{tab:main} complements this table: it measures
state carry across adjacent manifest windows, while the 32/64-frame rows here
stress one longer teacher window at a time.

The mask result is not a claim that AutoGaze is universally better than patch

selected indices to the host for mask construction, while AutoGaze-style
precomputed masks enter the JEPA pipeline directly as sparse token indices.

\section{Analysis and Reporting Practices}

The experiments support a narrower conclusion than ``sparse TTT solves
streaming video.''  They show that the implementation is numerically faithful,
that sparse encoder-side TTT can improve deployable free-run features, and
that long-horizon state drift is a trainable failure mode rather than an
unavoidable property of the adapter.  They also show that sparse inference and
sparse training have different bottlenecks: pixel-skip patchification helps the
forward path, while adapter backward and AdamW still dominate training.

Several reporting rules follow from this.  Real V-JEPA 2.1 parity must precede
quality claims because synthetic fixtures do not catch all positional encoding,
prefix mapping, and predictor-shape mistakes.  Free-run metrics must remain
separate from teacher-forced diagnostics because only free-run matches
deployment.  Sparse benchmarks must state whether masked pixels are skipped
before patchification or only gathered after dense patch embedding.  Temporal
adapters need state-health diagnostics because average 16-frame loss can hide
64-frame drift.  Backend labels should avoid overstating independence: Burn
\texttt{webgpu}, Burn native \texttt{wgpu}, and \texttt{burn\_flex\_gmm} WGPU
kernels are related backend-family lanes with different feature gates and
kernel surfaces.

\section{Limitations}

This is not yet a production-grade trained model.  The largest quality run is
small compared with open-set video pretraining, and the strongest AutoGaze
evidence still needs broader real-mask evaluation across domains.  The TBPTT
run greatly improves 32--64 frame stability, but it has not been validated on
hundreds or thousands of streamed frames, and the best diagnostic eval uses
small batch size because CUDA utilization probes can exhaust allocator pages at
batch 16.  The sparse training path is functional, but training throughput is
still limited by adapter backward and optimizer work.  The next production
gates are larger open-set training, exact upstream parity fixtures for both
base and large V-JEPA 2.1 variants, real AutoGaze masks, longer streamed
rollout validation, and lower-overhead TTT backward kernels.  We also do not
yet report downstream task accuracy; all quality numbers here are
teacher-feature distillation metrics.

\section{Conclusion}

Sparse V-JEPA 2.1 with encoder-side TTT is a viable direction for efficient
temporal video features, but the current checkpoint is not a final streaming
model.  The implementation is numerically aligned with the official V-JEPA 2.1
model, sparse CUDA inference is fast, and sparse free-run TTT improves over a
zero-init adapter baseline.  The first long-rollout ablation exposed fast-state
decoherence; the TBPTT stabilization run shows that carried-state training,
decay, regularization, and scheduled resets can make 32--64 frame rollouts
stable.  The path forward is therefore specific: keep production defaults
encoder-side, scale real-mask video training and streamed eval, and attack
backward/optimizer cost.
